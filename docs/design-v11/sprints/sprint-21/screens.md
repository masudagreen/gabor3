# Sprint 21 — v1.1.2 全ゲームのボタン撤去とガボール直接選択統一（実機フィードバック反映）

## スプリントの目的（spec-v11.md §13 Sprint 21）

1. ガボール直接選択が可能なゲーム（G-03 / G-04 / G-05 / G-06）で補助テキスト選択肢ボタン（horizontal-2 / clock-8）が画面から完全に撤去され、ユーザーは刺激領域内のガボールパッチを直接タップして回答する
2. G-09 / G-10 / G-11 / G-12 / G-13 については Designer の最終判断で UI 構造（直接選択化 or ボタン UI 維持しつつ空間対応配置）が本書に明記される
3. 全ゲームで結果オーバーレイ ◯/✕ がユーザー操作対象（直接選択ならガボール／象限、ボタン UI ならボタン）の上に正しく重畳される
4. 永続化スキーマ・staircase 値・閾値計算ロジック・採点ロジックは不変

含む機能：F-07（共通フォーマット改訂：直接選択原則の追加）、F-10（受け入れ基準補強：操作対象上の重畳）、§7.3 G-03 / §7.4 G-04 / §7.5 G-05 / §7.6 G-06（テキストボタン撤去 → ガボール直接選択）、§7.9 G-09 / §7.10 G-10 / §7.11 G-11 / §7.12 G-12 / §7.13 G-13（Designer 判断で UI 構造を確定）

### 2.2 Designer 判断でボタン UI 維持（3 ゲーム）

| ゲーム | 旧構造 | v1.1.2 新構造 | 採用案 | 採用理由 |
|---|---|---|---|---|
| G-09 | 横一列「flanker / target / flanker」+ 下部 horizontal-2「縦寄り／横寄り」 | 横一列「flanker / target / flanker」（target 直下に horizontal-2 を空間配置） | **案 B**：horizontal-2 維持 | Polat 2004/2012 パラダイム本来の「target 1 個 + flanker 2 個」構造を維持して臨床根拠を保つ。target に向き 2 択を重畳すると刺激の解像度・コントラストが乱れて Polat 系のクラウディング効果に干渉する。代わりに target 直下にボタンを空間配置することで、◯/✕ がユーザー操作対象上で成立する |
| G-12 | 中央 target + 周囲 4〜6 flanker + 下部 horizontal-4「垂直／水平／斜め右／斜め左」アイコン | 中央 target + 周囲 4〜6 flanker（target 直下に horizontal-4 を空間配置） | **案 B**：horizontal-4 維持 | target の向きは「抽象的な属性」で刺激上で直接選べない。重畳アイコン案（案 A）は注視中の target に余分な視覚要素を加えてクラウディング効果を希釈する懸念がある。target 直下に小さなボタン群（48px、注視を逸らさない最小サイズ）を配置することで、注視継続を妨げず空間対応も担保 |
| G-13 | ノイズ + 微小コントラスト数字 + 下部 keypad-10（5×2 = 10 ボタン） | ノイズ + 微小コントラスト数字（**刺激画像直下に keypad-10 を 5×2 で空間配置、刺激領域とボタン領域の中心軸を揃える**） | **案 B**：keypad-10 維持 | 0〜9 の数字 10 個を刺激上で直接選択させる構造は不可能（数字は埋め込まれた 1 個のみ）。keypad は数字選択 UI として最も慣用的・直感的（電話・電卓パラダイム）。配置は spec §7.13 推奨の 5×2 を採用（横スペース効率と縦コンパクト性のバランス） |

### 2.3 採用案の俯瞰 — 直接選択 vs ボタン維持の判断軸

| 判断軸 | 直接選択化を選んだ場合 | ボタン維持を選んだ場合 |
|---|---|---|
| 刺激本来の知覚パラダイム保存 | パッチ自体が選択肢になっても本質に影響しない（G-03 / G-04 / G-05 / G-06 / G-10 / G-11） | 本質に影響する場合は維持（G-09 Polat / G-12 Crowding は target 1 個構造が要） |
| 選択肢の意味と刺激領域の対応 | パッチ／象限自体が「答えの実体」（G-03 odd one、G-04〜G-06 の比較対象、G-10 の異向き象限） | 抽象属性の判定（G-09 縦／横、G-12 4 向き）や離散値選択（G-13 数字 10 種）は刺激上に重畳困難 |
| ◯/✕ 重畳の整合性 | パッチ／象限上に直接重畳。「自分が選んだ場所と結果が空間的に対応」が成立 | ボタン上に重畳。ボタンが target 直下に空間配置されていれば対応関係を視覚で追える |
| 注視継続への影響 | 直接タップは注視と並行可能（特に G-04〜G-06 / G-11 は 2 択） | target 直下のボタン群は視線移動を最小限に抑える配置が必要（G-09 / G-12 / G-13） |

## 7. S21-G09-PLAY：G-09 プレイ画面（horizontal-2 維持、案 B 採用）

### 7.1 Designer 判断と理由

**採用案：案 B（horizontal-2 維持、target 直下に空間配置）**

#### なぜ案 A（target 2 個化）を採らないか

- Polat 2004/2012 の Lateral Masking パラダイムは「中央 1 個の閾値付近 target + 両側 flanker」の構造そのものが訓練効果のエビデンス源。target を 2 個に増やすと：
  - target 間に「flanker」のような相互干渉が生じる
  - 両 target に flanker をそれぞれ配置すると刺激密度が変わり、staircase 値（target コントラスト + flanker 距離）の臨床根拠が失われる
- spec §7.9 にも「Polat パラダイムの臨床根拠を維持」を推奨案として明記

#### なぜ案 B が成立するか

- horizontal-2 ボタンを target の **直下**（数 px のクリアランス）に空間配置することで、ユーザー操作対象（ボタン）と刺激領域（target）が視線移動 1〜2°以内で対応づく
- 結果オーバーレイ ◯/✕ はボタン上に重畳されるが、target からボタンへの視線移動が小さいため、UX 一貫性は損なわれない

### 10.1 Designer 判断と理由

**採用案：案 B（horizontal-4 維持、target 直下に空間配置）**

#### なぜ案 A（target 周辺に向きアイコン重畳）を採らないか

- Crowding パラダイムは「target 周囲の flanker による視覚的干渉」を 60 秒注視で体験する課題。target 周辺に追加の視覚要素（向きアイコン）を重畳すると：
  - flanker と新追加要素が混じり、target-flanker spacing の臨床的意味が変質
  - 注視中に target に重畳された向きアイコンが見えてしまうとクラウディング効果が希釈される
- spec §7.12 注：「Crowding パラダイムの『target を見つめ続ける注視』を妨げない」ことを推奨

#### なぜ案 B が成立するか

- 4 択ボタン（垂直／水平／斜め右／斜め左）を **target の真下**（target 中心軸の直下）に配置することで、注視を逸らさずに済む
- ボタン群を小さめ（48〜56px）にし、target との距離を最小（space.4 = 16px）に保つことで視線移動を抑制
- 結果オーバーレイ ◯/✕ がボタン上に重畳されることで「自分が選んだ向きが正解か」が空間的に分かる

- 数字 0〜9 の 10 種を「刺激上で直接選択」させる構造が物理的に不可能（数字は刺激上に埋め込まれた 1 個のみ。10 個全部を表示する余地はない）

  - 5×2 はスマホ縦・PC 横どちらでも刺激領域の幅にほぼ収まり、刺激領域とボタン領域の中心軸を揃えやすい

- **G-03 / G-04 / G-05 / G-06 / G-10 / G-11**：プレイ画面に `AnswerChoiceGroup` / `ClockChoiceLayout` 由来のボタン要素が **存在しない** ことをテストで保証（querySelectorAll('button[role="radio"][data-from="answer-choice-group"]') == 0 等）

  - target 中心 X 座標 ≒ ボタン群中心 X 座標（許容差 ±8px）
  - ボタン上端 Y 座標 - 刺激領域下端 Y 座標 ≦ 16 + α px

### 16.3 ◯/✕ 重畳

- **直接選択ゲーム**：result phase で `data-target-id` 経由で正解パッチ／象限上に MarkBadge `kind="correct"` が、ユーザー誤選択パッチ／象限上に `kind="wrong"` が重畳される
- **ボタン維持ゲーム**：同じく result phase で対応するボタン上に MarkBadge が重畳される
- 偶然正解時は ◯ のみ、✕ なし
- スクリーンリーダー読み上げが「正解です／不正解です／正解ですが選ばれませんでした」のいずれかで適切

### 16.4 staircase / 採点ロジック不変性

- 直接選択化されたゲーム（G-03 / G-04 / G-05 / G-06 / G-10 / G-11）の staircase 値・採点ロジック・閾値計算が v1.1（Sprint 11〜16）と完全一致
- 永続化スキーマ（spec §11 / §F-09 系）も不変
- リグレッションテスト：旧プレイ画面と同じ入力（パッチ選択）で同じ採点結果と staircase 推移

### 16.5 a11y

- `role="radiogroup"`, `role="radio"`, `aria-checked` の適切な付与
- ガボールセルが SR から到達可能（`tabindex="0"` または radiogroup ナビゲーション可）
- guidance テキストの `aria-live="polite"` 読み上げ（出題方向切替時）
- ResultOverlay の `aria-live="assertive"` 読み上げ（採点結果）
- 矢印キーナビゲーション（G-03 円周、G-10 象限、G-04〜G-06 / G-11 左右）
- focus-visible 3px 青 outline

- ボタン維持ゲーム（G-09 / G-12 / G-13）でボタンが刺激領域直下に空間対応配置（中心軸一致）
