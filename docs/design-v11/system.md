## 1. v1.1 で新規・改訂・削除する設計規範

### 1.1 OPT-12 全ゲーム共通注視フォーマット（新規・最重要）

13 ゲーム全てが以下の画面骨格に従います。これは v1 デザインシステムの §0「老眼最適化原則」に追加される **OPT-12** に相当する規範です。

```
┌──────────────────────────────────────────────┐
│  ✕    残り N 秒                              │ ← GameStatusBar (v1.1)
│                                              │   左：閉じる × (48pt)
│                                              │   中：font.h2 30px Bold tabular-nums
│                                              │   右：何も置かない（試行数表記廃止）
│  ┌────────────────────────────────────────┐  │
│  │                                        │  │
│  │      [ ゲーム固有の注視領域 ]           │  │ ← 60 秒間ずっと表示
│  │      （点滅・切替・マスクなし）         │  │   gabor.frame.padding=24
│  │                                        │  │   背景は #808080 固定
│  │                                        │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐    │ ← 選択肢ボタン群
│  │ 選択肢 A  │  │ 選択肢 B  │  │ 選択肢 C  │    │   AnswerChoiceGroup
│  │ (ハイライト│  │           │  │           │    │   選択中＝黄色 4px 枠
│  │  4px 黄)  │  │           │  │           │    │   タップで切替
│  └──────────┘  └──────────┘  └──────────┘    │   再タップで解除
│                                              │
└──────────────────────────────────────────────┘
```

#### 共通骨格の遵守規範
- **画面上部固定領域（高 64px）**：左に × ボタン（IconButton lg、ariaLabel="ゲームを中断"）、中央に「残り N 秒」（`font.h2` 30px Bold tabular-nums）。**v1 で右側に表示していた「N / M 試行」は廃止**（OPT-12：1 セッション 1 試行になったため）
- **中央注視領域**：ゲーム固有のガボール／グリッド／パッチを 60 秒間「点滅なし・切替なし・フェードなし・マスクなし・モーフィング許容（G-01 のみ）」で表示
- **下部選択肢群**：`AnswerChoiceGroup`（v1.1 新規コンポーネント、`components.md` §AC-1 参照）。タップで選択トグル、別を押すと切替、再タップで解除
- **「現在の回答：◯◯」のテキスト表示は置かない**（Q5 確定）。選択中はボタン／画像自体が枠でハイライトされる
- **試行中の正誤フィードバックは出さない**（注視を妨げない）。F-10 結果オーバーレイで開示

> **v1.1.1 改訂（Sprint 20）**：v1.1 では「黄色 4px 枠」をハイライトとしていたが、実機テストで「黄色 4px 枠が目立ちすぎてガボールが見えづらい」フィードバックを受け、**控えめな選択枠（2px 以下、ガボール視認性を阻害しない中性色）** に変更した。具体値は `components.md` §4 `ImageChoiceCell` および §3 `AnswerChoiceGroup` で定義する。新トークン `color.selection.subtle` を導入（後述）。

> **v1.1.2 改訂（Sprint 21）**：選択肢領域は「**刺激領域内の要素（ガボール／象限）を直接タップ**」を原則とし、上記 OPT-12 の骨格における `下部選択肢ボタン群` 領域は **G-09 / G-12 / G-13 でのみ表示** され、それ以外のゲームでは選択肢ボタンは存在しない（パッチ自体が ImageChoiceCell でラップされて選択肢になる）。詳細は `sprints/sprint-21/screens.md` 参照。直接選択原則化に伴い「ガボール領域は装飾扱いで `aria-hidden="true"`」の規約は撤回し、ImageChoiceCell でラップされたガボールは radio として SR にも到達可能にする（具体は components.md §4 a11y 節）。

- **未回答のまま 60 秒経過は「不正解」**として記録、staircase は易方向に動く
- **早期終了不可（OPT-11）**。中断は × ボタンのみ（AbortConfirmDialog 経由）

#### 画面分割の数値規範（スマホ 375×667 基準）

| 領域 | 高さ | 幅 |
|---|---|---|
| GameStatusBar | 64px | 100% |
| 注視領域パディング | space.4（16px）天地 | space.4 左右 |
| 注視領域コンテンツ | 280〜320px（ゲームごとに最適化、`min(viewport.width-32, 360)`） | 同左 |
| 選択肢領域パディング | space.5（24px）上、space.6（32px）下 | space.4（16px）左右 |
| 選択肢ボタン | 64px（lg） | flex-1 / 各 minWidth 96px |

#### PC 横 1280×800

中央 720px 最大幅、ガボール領域は最大 480×480px（v1 と同じ）。選択肢は注視領域の下に水平に並べる。注視領域脇に置かない（読み上げ順序の都合）。

#### a11y
- 選択中の選択肢は `aria-checked="true"`（ラジオグループ）または `aria-pressed="true"`（トグル単体）
- カウントダウンは `aria-live="polite"`、5 秒以下 polite、6 秒以上 off に切替（v1 と同じ）
- ガボール領域は装飾扱いで `aria-hidden="true"`、ARIA テキストとしてゲーム固有のガイド（「左右 2 つの縞模様のうち、どちらが時計回りに傾いているかを選んでください」）を `aria-describedby` で読ませる

### 1.3 F-10 結果サマリの統一構造（v1.1 改訂、最重要 / v1.1.1 で再改訂）

v1 では Game 1/2/3 ごとに結果サマリのレイアウトが異なっていました。v1.1 では **13 ゲーム全部が以下の 1 枚レイアウトに統一** されました。

> **v1.1.1 改訂（Sprint 20）**：v1.1 までは独立した結果画面に遷移していたが、実機テストで「文字だけでは自分が画面のどこを選んだのか視覚的に対応づけられない」フィードバックを受け、**結果開示を刺激画面に統合**する方式に再改訂した。本 §1.3 の枠囲みレイアウトは「v1.1 期の旧レイアウト」として記録目的で残し、v1.1.1 以降の最新規範は **§1.3.1 結果オーバーレイ統合方式** で上書きする（後述）。
> Sprint 20 の詳細は `docs/design-v11/sprints/sprint-20/screens.md` を参照。

```
┌─────────────────────────────────────┐
│              G-04 の結果             │ ← font.h2 30px Bold（ゲーム名）
│                                     │
│         正解は 「右が濃い」          │ ← font.h1 36px Bold + 黄色装飾
│                                     │
│         あなたの回答 「左が濃い」    │ ← font.body.lg 26px Medium
│                                     │   不正解時：error 色装飾アイコン併記
│                                     │
│  ┌───────────────┐ ┌───────────────┐│
│  │ 今回の閾値     │ │ 前回比         ││ ← 2 列メトリクスカード
│  │               │ │               ││   metric-card .value
│  │   0.12       │ │   +0.03 ↑     ││   font.h2 30px Bold
│  │ コントラスト差 │ │  改善         ││   ラベル font.body 24px
│  └───────────────┘ └───────────────┘│
│                                     │
│  ┌───────────────────────────────┐  │
│  │     次へ                      │  │ ← Primary Button (lg, 64px)
│  └───────────────────────────────┘  │   コース時：右上に 10 秒カウントダウン併記
│                                     │   単体時：自動進行なし
└─────────────────────────────────────┘
```

#### 規範
- 「正解は ◯◯」を最上段に大きく表示（`font.h1` 36px Bold + 黄色 4px 枠装飾、点滅なし）
- 「あなたの回答 ◯◯」を 2 段目（`font.body.lg` 26px Medium）。不正解時は文字色は `color.fg.primary`、装飾アイコンのみ `palette.semantic.error`（§1.4 制約に従い、テキスト色には使わない）
- 「今回の閾値」と「前回比」を 2 列カード（`MetricCard` v1.1 新規、`components.md` §MC-1 参照）。値は `font.h2` 30px Bold（数値強調）、ラベルは `font.body` 24px
- 前回比の direction（up/down/flat）は装飾矢印アイコン（success / warning / muted 色）+ テキスト「改善」「同等」「やや低下」で表現。色のみで判別させない（NF-12）
- 単体プレイ時は「次へ」ボタンを押すまで進まない（OPT-7）
- コースモード時は「次へ」ボタンの右上に **10 秒カウントダウン**（`font.h3` 26px、CountdownDisplay md）を併記し、ユーザー操作なしで自動進行
- バッジ獲得時はサマリ画面下部に AchievementBadge 獲得演出（v1 同仕様）を 1.5 秒挿入

#### 比較演出
- 「+0.03 改善」のような前回比は、v1.1 staircase 改訂（直近 5 セッション平均）に従い、前回までの 5 セッション平均 vs 今回値の差分で計算
- データが 1 件目（前回なし）のときは「初回測定」と表示、矢印は表示しない

### 1.3.1 結果オーバーレイ統合方式（v1.1.1 新規・最重要、F-10 再改訂、v1.1.2 で全ゲーム直接選択原則を追加）

Sprint 20 で導入。13 ゲーム共通で、60 秒経過後は刺激画面に**とどまったまま**結果を開示する。独立した結果画面（旧 `ResultSummaryV11` の独立画面・旧 `CourseInterstitialResultScreen`）への遷移は廃止。

> **v1.1.1 仕様再改訂（Sprint 20、spec 再確定）**：実機テストの追フィードバックで「結果オーバーレイ上のメトリクスバー（閾値・前回比・単位）が刺激領域の視認を妨げる」との指摘を受け、**メトリクスバーは ResultOverlay から完全撤去**した。閾値の数値推移は **F-11 進捗グラフ**で参照する設計とする。結果オーバーレイの構成は **◯ / ✕ アイコン + 『次へ』ボタン（およびコースモード時のカウントダウン）のみ**となる。

> **v1.1.2 仕様再々改訂（Sprint 21、2026-05-01）**：実機テストで「他のゲームもガボールの下にボタンの選択肢が残っているのが煩わしい／回答表示が直感的でない」とのフィードバックを受け、**全ゲームでガボール／刺激領域を直接タップして回答する形を原則化**する。これに伴い、結果オーバーレイの ◯/✕ 重畳位置も **「ユーザー操作対象（直接選択ゲームならガボール／象限、ボタン維持ゲームならボタン）の中央」に統一**する（v1.1.1 と同方針だが対象範囲が広がる）。直接選択化したゲーム（G-03 / G-04 / G-05 / G-06 / G-10 / G-11）では刺激領域内のガボール／象限上に、Designer 判断でボタン UI を残したゲーム（G-09 / G-12 / G-13）ではボタン上に ◯/✕ を重畳する。

```
┌─────────────────────────────────────┐
│  ✕    残り 0 秒（停止）              │ ← GameStatusBarV11 そのまま
│                                     │   ✕ は維持（ホーム強制脱出用）
│  ┌────────────────────────────────┐ │

│  │   ┌──────┐    ┌──────┐         │ │ ← 注視領域はそのまま残す

│  │   │  ◯   │    │  ✕   │         │ │   各パッチ／領域の中央に
│  │   └──────┘    └──────┘         │ │   ◯ / ✕ アイコンを重畳表示
│  │                                │ │   （半透明背景 + アイコン）
│  └────────────────────────────────┘ │
│                                     │
│  ┌─────────────────────────────────┐│
│  │     次へ          ⏱ 8           ││ ← Primary lg 64px
│  └─────────────────────────────────┘│   コース時のみ右側にカウントダウン併記
└─────────────────────────────────────┘
```

#### 規範

1. **画面遷移しない**：60 秒経過後、`GamePlaySurface` の `stimulusArea` はそのまま固定表示し、`answerChoices` 領域のみが「『次へ』ボタン」に置き換わる。`stimulusArea` 内の各選択対象（パッチ／グリッドセル／領域）の上に **◯ / ✕ アイコンレイヤー**を重畳する
2. **◯ アイコン**：採点における「正解」位置（出題者が指定した正解側）に表示。色は `palette.semantic.success`（緑系）、半透明白背景の上に配置
3. **✕ アイコン**：「ユーザーが選択したが不正解だった」位置のみに表示。色は `palette.semantic.error`（赤系）、半透明白背景の上に配置
4. **正解と一致して選んだ位置**：◯ のみ表示（✕ は付かない）。✕ + ◯ の同時重畳は発生しない
5. **複数選択ゲーム（G-01 / G-07 / G-09）**：各パッチ単位で個別に ◯ / ✕ を判定して重畳。「正解だが選ばれなかった」パッチには ◯ を表示するが、不透明度を 50% 落として「取りこぼし」を視覚的に区別する（薄 ◯）。「不正解で選ばれなかった」パッチには何も表示しない
6. **メトリクスバー（閾値・前回比・単位）は表示しない**：刺激の視認を妨げる追加メトリクスは ResultOverlay には載せない。閾値の数値や時系列推移は F-11 進捗グラフから参照する設計とする
7. **「次へ」ボタン + カウントダウン**：刺激領域の直下に Primary Button lg（64px、`font.body.lg` 26px Bold）を配置。コース時は右内側にカウントダウン（`font.h3` 26px tabular-nums）を併記、10 秒で自動進行。単体時はカウントダウンなしで押すまで停止
8. **試行中（注視 60 秒間）は ◯/✕ オーバーレイを表示しない**（F-07 と矛盾しない）
9. **オーバーレイ表示中は `stimulusArea` 内の選択肢タップは無効**（採点後の誤操作防止）。`pointer-events: none` を `stimulusArea` 全体に適用
10. **アニメ・点滅は使わない**（OPT-9 / NF-11）。フェードイン 200ms（`prefers-reduced-motion: reduce` 時は 0ms）でオーバーレイが現れる
11. **バッジ獲得演出**（F-13、1.5 秒）はオーバーレイの上層レイヤで 1 度だけ流れる（位置は画面中央上、「次へ」ボタンと干渉しない）

#### a11y

- 60 秒経過時に `aria-live="assertive"` で結果文字列を 1 度読み上げ（例：「G-04 結果。正解は右が濃い。あなたの回答は左が濃い。次へボタン」）
- ◯ / ✕ アイコンには `aria-label="正解です" / "不正解です"` を付与
- 「次へ」ボタンに自動フォーカス（カウントダウンは継続）
- 単体時は「次へ」のすぐ下に `SinglePlayPostFooter`（同じゲームをもう一度／一覧へ／ホームへ）が表示されるため、フッター用の縦スクロールを許容する。コース時はフッターを表示しない

#### 旧 §1.3 レイアウトとの関係

- 旧 §1.3（独立画面の `ResultSummaryV11`）は v1.1.1 では撤去。`ResultSummaryV11` が表示していた閾値メトリクスは ResultOverlay には**引き継がれない**（F-11 進捗グラフで参照する設計）
- `MetricCard` も ResultOverlay 内では使用しない（メトリクスバー自体が撤去されたため）
- `SinglePlayPostFooter`（FT-1）は本オーバーレイの「次へ」ボタンの直下に縦に展開される（単体時のみ）

v1 では「準備ができました」ボタンを押して進行していましたが、**v1.1 では 3 秒カウントダウンで自動進行**（Q11 確定）。

```
┌─────────────────────────────────────┐
│  ✕                                  │ ← 緊急脱出のみ（タップで戻る）
```

## 3. F-18 `releaseEnabled` 動的除外の設計上の保証

仕様 F-18 は「`releaseEnabled=false` のゲームをホーム / コース / グラフ / バッジから動的除外する」を要求します。設計上、以下の規範で**矛盾発生を抑止**します。

### 3.1 単一ソース：gameRegistry
- 13 ゲームの定義（ID / 表示名 / 系統 / `releaseEnabled` / `order`）は `gameRegistry` に集約
- 全画面・全コンポーネントは `gameRegistry.filter(g => g.releaseEnabled)` を経由してゲーム一覧を取得する
- **直接ゲーム ID 配列をハードコードしない**（screens.md でも `[G-01, G-02, ...]` と書かず、「enabled な全ゲーム」と表現する）

### 3.2 反映ルール

| 画面 / 機能 | 反映ルール |
|---|---|
| ホーム CTA「全ゲーム連続プレイ（約 N 分）」 | N = enabled ゲーム数。CTA ラベルが動的計算 |
| 全ゲーム一覧（単体プレイ） | enabled ゲームのみ表示。Card 並びは `order` 昇順 |
| 全ゲーム連続コース | enabled ゲームのみが対象。順序は `order` 昇順 or 日付シードランダム |
| 進捗グラフ「ワイドスコア」 | enabled ゲームの正規化スコア平均で算出 |
| 進捗グラフ「ゲーム別」子タブ | enabled ゲームのみタブとして表示 |
| バッジ一覧（B-01〜B-13） | バッジ自体は全 13 種を一覧表示するが、獲得条件評価は enabled ゲーム集合で再評価。disabled に依存するバッジは「獲得条件 × 該当ゲーム除外で評価」と注記表示（後述） |
| ゲーム単体起動の URL ／ DeepLink | disabled ゲームへのアクセスは「現在このゲームは公開対象外です」を表示してホームに戻す |

### 3.3 バッジの動的除外文言

`releaseEnabled=false` のゲームに依存するバッジ（B-06、B-07、B-08、B-09、B-10、B-13）は、disabled ゲームを除外した集合で評価し、未獲得時のヒントには
> 「現在公開対象の N ゲーム全てで条件を満たすと獲得」

のように動的な N を表示します。

### 3.4 設計上のチェック手順
- Designer は 13 ゲーム分のモックを作成するが、**画面のレイアウトは「ゲーム数が 7〜13 のいずれでも崩れない」前提**で設計
- 横スクロールタブは折り返しでなくスクロール可。グリッドは 2〜4 列で自動列数調整
- 結果サマリ「次へ」ボタン押下時、コースの最終ゲームかどうかは `currentIndex === enabledGames.length - 1` で判定（13 固定でない）

## 1.12 ガボールクリッピング品質規範（NF-27 / NF-28、v1.2 新規）

### 1.12.1 描画方式（Designer 規範）

「**実サイズの N 倍で生成 → 矩形枠でクリッピング**」方式を採用する。

- **N = 1.5（Designer 提案値）**：表示する見かけサイズ（例 240px 角）に対し、内部生成は 1.5 倍（360px 角）で行う。ガウス窓の裾野は実サイズ角の外まで連続的に伸びている状態で、表示時に 240px の矩形枠でクリップする
- これにより：
  - 角度を最小単位（NF-26）ずつ変えても、パッチ枠と縞模様の間に「背景色（グレー）が見える」隙間が出ない
  - 矩形枠は描画上の clipping mask（`overflow: hidden` + `width/height` 固定）で実現
  - ガウス窓の裾野が矩形端部まで連続するため、縞模様の途切れが矩形辺と一致せず、自然なフェードアウトが見える

### 1.12.2 N の根拠
- N = 1.0：枠ぴったりに生成 → 角度回転時に四隅で背景色が露出
- N = 1.4（√2）：90° 回転までは保証されるが、それ以下の角度（最小単位 0.25° など）でわずかに四隅が見える可能性
- **N = 1.5（採用）**：完全な保証 + 実装の余裕。Generator が 1.4 で十分と判断する場合は妥協可能だが、本書では 1.5 を Designer 提案値とする
- N が大きいほど描画コストが増える（240px → 360px は 2.25 倍のピクセル数）。GPU 描画の場合は問題ないが、CPU 描画時はパフォーマンス劣化の可能性。Generator が `react-native-skia` 等のハードウェア描画を採用していれば 1.5 で問題ない

### 1.12.3 受け入れ基準（Designer のチェック）
- 角度を最小単位（NF-26 / §1.13）ずつ 0° → 360° まで動かしてスクリーンショットを取り、四隅で背景色（#808080）が露出していないことを目視確認
- 静止画（角度固定）でも同様に確認
- 円形マスクではなく矩形マスクを採用する根拠：複数パッチをグリッド配置するときに矩形のほうがレイアウトが安定し、隙間トラッキングが容易

## 1.13 角度刻み Designer 提案値（NF-26、v1.2 新規）

spec NF-26 では「Designer/Generator が表示サイズと cpd から導出した最小可視差を採用する」とされる。本節で具体値を提案する。

### 1.13.1 最小可視差テーブル（Designer 提案値）

最小可視差 = ユーザーが「向きの違い」を弁別できる最小角度差。表示サイズが大きいほど・cpd が高いほど（縞が細かいほど）、より小さい角度差を弁別できる。

| 表示サイズ（パッチ 1 個の辺長） | 1.5 cpd | 3 cpd | 6 cpd | 9 cpd |
|---|---|---|---|---|
| **80px**（G-09 等の小サイズ） | 5° | 3° | 2° | 1.5° |
| **100px**（G-11 等） | 4° | 2.5° | 1.5° | 1° |
| **120px**（G-04〜G-06 等の中サイズ、3×3 グリッド内） | 3° | 2° | 1° | 0.5° |
| **160px**（G-04〜G-06 PC 横） | 2° | 1° | 0.5° | 0.25° |
| **200px**（G-13 数字探しの個別ノイズパッチ） | 2° | 1° | 0.5° | 0.25° |
| **240px**（単発表示等） | 1.5° | 1° | 0.5° | 0.25° |
| **320px〜480px**（G-13 全体表示） | 1° | 0.5° | 0.25° | 0.1° |

### 1.13.2 算出根拠（Designer メモ）

- ヒトの方位弁別閾値は 6 cpd 以上の刺激で 1° 前後（V1 領域の方位選択細胞の帯域幅由来）
- ガボールパッチ表示時は、パッチサイズが小さいほど縞の本数が減り、向きの判定が困難になる
- 表示サイズ × cpd で「画面上の縞本数」が決まる。例：120px × 6 cpd ＠ 40cm 視聴 ≒ 3〜4 本の縞 → 1° 弁別可能
- 上表は「中央〜難度寄り」の初期値ではなく、staircase が到達しうる**最も難しい側の刻み単位**を示す

### 1.13.3 staircase との連携

- staircase の step 単位は本テーブルの最小可視差をベースに決める：
  - 例：G-04 contrast の場合は contrast 差で staircase するため本テーブル直接適用ではないが、表示時の cpd は本テーブルから「縞の見え方が安定する範囲」を選ぶ
  - 例：G-07 向き同定の場合は「向きズレ許容角」が staircase パラメータ。step は本テーブルの最小可視差の 2 倍程度を初期値とする（120px × 6 cpd → step 2°、最小到達値 1°）
- 角度の絶対値は spec.md には書かない（NF-26 通り）。本書で Designer が技術判断として明記する

### 1.13.4 表示サイズの実測手段（Generator 連携）

- 表示サイズは「グリッドセルサイズ」または「単発パッチサイズ」を実測（CSS から px 取得）
- cpd は `cpd ≒ 縞本数 / (表示サイズ × tan(視野角)/π × 180)` で逆算可能だが、本アプリでは spec の cpd プリセット（1.5 / 3 / 6 / 9）を使うため、staircase 実装側で「現在のサイズと cpd ペアから最小可視差を引く」テーブルを参照する

## 1.14 ステータスバー回避規範（NF-29 / NF-30、v1.2 新規）

### 1.14.1 「ゲーム画面はフルスクリーン許容」の解釈

- **ゲーム画面**（G-01・G-03・G-04・G-05・G-06・G-07・G-13 のプレイ中の刺激提示画面）はステータスバー領域も使用してよい
- 理由：刺激の視認領域を最大化するため。ガボール背景（#808080）がステータスバー領域まで自然に広がる方が没入感が増す
- 上部 GameStatusBarV12（× ボタン + 残り秒数）はステータスバー高さ分だけ下にオフセット（iOS = SafeAreaInset.top、Android = StatusBar.currentHeight）。GameStatusBarV12 の背景は `color.bg.surface`（半透明 80% でもよい）でステータスバー領域と滑らかに接続

### 1.14.2 「それ以外はセーフエリア準拠」の解釈

- 非ゲーム画面（F-01 オンボーディング / F-16 距離リマインド / F-15 クールダウン / F-21 事後画面 / 設定 / 進捗グラフ / バッジ）はセーフエリアの内側に全コンテンツを配置
- iOS：`SafeAreaView` または `useSafeAreaInsets` で top / bottom / left / right の inset を取得し、ルート要素の padding に適用
- Android：`StatusBar.currentHeight` で上部高さ取得 + 下部 navigation bar 高さも同様
- Web：CSS `env(safe-area-inset-top)` を viewport-fit=cover と組み合わせ

### 1.14.3 SafeAreaWrapper コンポーネント（components.md §31 で詳述）

- 全非ゲーム画面で必ず使うラッパー
- `mode: "game"` 時はステータスバー領域も使用許容（top inset を無視、bottom と horizontal は適用）
- `mode: "default"` 時は全 inset を適用

### 1.15.1 イベント別の発火パターン

### 1.15.2 サイレントモード時の挙動（NF-33）

- iOS：システムの「サイレントモード」（ベルアイコン）が ON のとき、音は再生されないが、ハプティクスは発火する（ユーザーが振動 OFF にしていない限り）
- Android：システムの「マナーモード（バイブのみ）」/「サイレントモード」に従う。バイブのみモードは音なし＋振動あり、完全サイレントは音も振動もなし（ただしハプティクスは「Vibrator.vibrate()」を呼ぶ実装で、ユーザー設定の振動 OFF とは別の OS 側設定）
- アプリ内設定の「音 ON/OFF」「振動 ON/OFF」は OS 設定よりさらに細かい制御として動作

### 1.15.3 実装抽象化（Generator 参考）

- `../rapidreading2/src/platform/audio.ts` の `playSound(key, opts)` および `triggerHaptics(pattern)` 抽象を流用

## 1.17 7 ゲーム共通グリッド配置規範（v1.2 新規）

A13 確定（数値希望なし、Designer 判断、隙間最小化）に基づく Designer 提案値。

### 1.17.1 グリッド隙間（Designer 提案値）

| グリッド | パッチ間隙間 | 理由 |
|---|---|---|
| 3×3（G-01 / G-04 / G-05 / G-06） | **8px**（space.3 相当の控えめな値） | 隙間最小化方針。パッチ同士の輝度干渉を避ける最小幅。3 列の場合、全体辺長 320px に対して 2 つの隙間 = 16px、パッチ 1 個 = (320-16)/3 = 約 101px |
| 4×4（G-07） | **6px**（space.2 相当） | 4 列のためパッチ 1 個サイズが小さくなる。隙間をさらに詰めて視認性確保。全体辺 320px に対して 3 つの隙間 = 18px、パッチ 1 個 = (320-18)/4 = 約 75px |

> **対応する v1.1 規範との関係**：v1.1 の §1.1 では「セル間ギャップ space.4（16px）」だったが、v1.2 では「ガボールクリッピング方式（§1.12）」を採用したため、隙間が背景色露出の原因にならず、もっと詰めて配置できる。よって 8px / 6px に縮小した。

### 1.17.2 グリッド全体サイズ

| デバイス | 3×3 全体辺 | 4×4 全体辺 |
|---|---|---|
| スマホ縦（375） | `min(viewport.width - 32, 320)px` | `min(viewport.width - 32, 320)px` |
| PC 横（1280） | 480px | 480px |

### 1.17.3 静止アイテムの角度差（NF-28b 関連、Designer 提案値）

G-01 において静止アイテム同士の角度差は「回転アイテムの 1 秒分の回転量より十分大きく」必要。

- 回転速度 staircase 範囲：1°/s（最難）〜 6°/s（最易）、初期 3°/s
- 1 秒分の回転量：1°〜6°
- **静止アイテム同士の最小角度差 = 10°（Designer 提案値）**：1 秒分の最大値 6° の 1.67 倍。ユーザーが「静止」と「ゆっくり回転」を見分けられる十分なマージン
- 静止アイテムは互いに 10° 以上離れたランダム角を選ぶ（例：8 個の静止アイテム × 9 セルから 1〜3 個の回転を引いた残り 6〜8 個に対し、互いに 10° 以上離れる）

### 1.17.4 G-01 回転速度 staircase 提案値（Designer）

- 易：6°/s（明らかに動いているのが分かる速度）
- 中央〜難度寄り（初期）：3°/s
- 難：1°/s（ぎりぎり「動いていそう」と感じる速度）
- step：0.5°/s
- 受け入れ基準：色のみで段階を伝えない原則と同様、回転は連続変化なので staircase の step だけ管理

### 1.17.5 G-01 回転アイテム個数分布（Designer 提案値）

- 1〜3 個ランダム
- 確率分布：1 個 40%、2 個 40%、3 個 20%
- 同セッション内では固定（試行ごとにランダム抽選）
- 採点 TP+1 / FP-1 / FN 0 において、ユーザーが 0 個選択 → 全 FN（スコア 0）、誤って全 9 セル選択 → TP+正解数 - FP+誤答数

### 1.17.6 G-04 / G-05 / G-06 違うパッチ個数分布（Designer 提案値）

- 1〜3 個ランダム
- 確率分布：1 個 35%、2 個 40%、3 個 25%
- 同セッション内では固定

### 1.17.7 G-04 / G-05 / G-06 「違いの強さ」初期値（Designer 提案値）

> **G-06 のみ v1.1 維持**：A3 確定。窓サイズ範囲は v1.1 値を維持する。

### 1.17.8 G-07 同じ向きアイテム個数分布（Designer 提案値）

- 4×4 = 16 セル中、同じ向きのアイテム：2〜5 個ランダム
