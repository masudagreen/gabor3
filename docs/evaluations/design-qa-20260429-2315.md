## 1. 総合判定

**❌ 不合格**

理由（要約）：
1. **system.md §1.4 のコントラスト比表に複数の数値偽装/誤計算**があり、`fg.onPrimary on action.primary (light)` の主張 7.6:1 は実測 5.93:1（AAA を満たさない）。`fg.secondary on bg.canvas (dark)` も主張 9.1:1 が実測 6.54:1。これは**システム自身が掲げる「コントラスト比 7:1 以上厳守」を満たさず、Designer の自己矛盾（=トークン整合性 Critical）**
2. **system.md 内で「18pt」と「18px」の混同**があり、§0 OPT-1（L11）/ §2.1（L112）では「18pt（24px）」と書きながら §2.2 タイプスケール表（L122-124）では `font.body = 18px` と定義。spec.md 明示の「18pt（≒24px）」基準を実装上満たせない。**全モックアップ 5 ファイルがこの矛盾の影響で本文・ボタン文字を 18px で実装しており、spec.md OPT-1 違反**
3. **Sprint 3 mockup の Game 3 時計回答ボタン 8 個全てが 16px フォント**（OPT-1: ボタン文字 18px 以上に違反、spec.md では 18pt=24px 以上）
4. **Sprint 2 mockup の Home スマホ 360px で Game カード 3 列レイアウトが破綻**：「変化察知」「二重表裏判別」「周辺視野ハント」のテキストが縦詰まりで読みづらい（レスポンシブ Major）

Critical 件数（独立な事象として）：3 件 / Major 件数: 4 件 / Minor: 3 件

## 2. 観点別評価

### 2.1 アクセシビリティ（a11y）

**判定: 不合格（Critical 2 件）**

#### Critical
- **[a11y / OPT-1] Game 3 mockup の時計回答ボタン 8 個が全て 16px フォント** — spec.md OPT-1 は「ボタン文字 18pt 以上」を強制（system.md §0 では 18pt=24px、§2.2 では 18pt=18px と内部矛盾だが、いずれにせよ 16px はどちらの解釈でも下回る）。`docs/design/sprints/sprint-3/mockups/game3-play.html` line 130 の `.clock-btn { font-size: 16px; }` が原因。8 個のボタン全部に影響。
- **[a11y / トークン整合性] system.md §1.4 のコントラスト比 4 件が実際の値より高めに記載されており（数値偽装または計算誤り）、システムが掲げる「7:1 以上を厳守（OPT-5 / NF-8）」を破っている**：
  - `fg.onPrimary (#FFFFFF) on action.primary (#1D5BD8)` 主張 **7.6:1** ⇒ 実測 **5.93:1**（AA 止まり）
  - `fg.secondary (#8A909A) on bg.canvas dark (#000000)` 主張 **9.1:1** ⇒ 実測 **6.54:1**（7:1 未満）
  - `semantic.error (#B6271F) on bg.surface light (#FFFFFF)` 主張 **7.1:1** ⇒ 実測 **6.37:1**（7:1 未満）
  - `fg.onPrimary on action.primaryHover` 主張 **9.8:1** ⇒ 実測 **8.61:1**（誤差小だが齟齬）
  - 加えて、`fg.muted (#6E7480)` on canvas は **4.50:1** で AAA 不合格、`streak.flame (#E07000)` on white は **3.24:1** で **AA すら不合格**。これらは screens.md / mockup 内で実際に使われているトークン（StreakBadge の炎色など）であり、実装すれば WCAG AA をも下回る。

#### Major
- **[a11y / OPT-1] system.md 内部の 18pt/18px 混同による 5 mockup 全部のボタン文字が 18-20px** — OPT-1 「ボタン文字 18pt 以上」= 24px 以上が spec.md の明示要件。20px は 18pt 強の換算（妥当解釈もあり得る）だが、18px は誤読。少なくとも **system.md §2.2 のタイポスケール表を spec.md と整合させる必要**がある（あるいは「18pt（≒18px）」と spec.md 解釈を Designer 主導で再定義し、ユーザー承認を取る）。
- **[a11y] Sprint 5 mockup `result-summary.html` の `metric-card .value` が 28px** — screens.md S5-01 表で「正答率 78% / 試行数 23」は `font.h3 22pt` と明示されているが、22pt = 29.33px。28px は若干小さい（誤差 1px、Minor）。

#### Minor
- **[a11y] Sprint 3 mockup `game3-play.html` で `class="patch odd"` という class 名が DOM にハードコード** — スクリーンリーダーは class を読まないので a11y 直接的影響はないが、JS Inspector で正解パッチを容易に判別できる。実装時は class 名/属性で odd one を識別できないようにする必要があり、screens.md S3-01 に「DOM 上でも判別不能化」の指示を追記するのが望ましい。
- **[a11y] system.md §17 アンチパターンに「14pt 以下のテキスト（OPT-1 違反）」と書いてあるが、OPT-1 の正式定義は「18pt 以上」**。14pt 以下を禁止しても 14-17pt の領域が許容されているように読める可能性。spec.md L13 と齟齬。

### 2.2 レスポンシブ

**判定: 条件付き合格（Major 1 件）**

- 全 5 mockup × 360 / 375 / 1280 / light / dark = 30 viewport で**横スクロール発生 0 件**（OK）
- 全 viewport でコンソールエラー 0 件（OK）

#### Major
- **[レスポンシブ] Sprint 2 mockup `home.html` の Game カード 3 列レイアウトが 360px で破綻** — `grid-template-columns: repeat(3, 1fr)` で各カード 106px（gap 12px 込み）。「変化察知」「二重表裏判別」「周辺視野ハント」のテキストが縦詰まりとなり、特に「周辺視野ハント」は縦並び 1 文字ずつになる。screens.md S2-01 のスマホレイアウト ASCII では `┌──────────┐ ┌──────────┐ ┌────┐` となっており、Game 3 カードが意図的に幅狭めだが、これは「Game 3 は準備中」だった暫定状態の図。Sprint 3 で Game 3 が有効化された時点で 3 等分割では破綻する設計。**スマホ縦では 2 列 + 折返し or アイコンサイズ縮小が必要**。

### 2.4 トークン整合性

**判定: 不合格（Critical 1 件）**

#### Critical
- **[トークン整合性] system.md 内で 18pt と 18px の混同**：
  - L11（OPT-1 表）: 「`font.body.min` = 18pt（24px）」
  - L112（§2.1 基本方針）: 「最低 18pt（24px）を本文の床として強制」
  - **L133（§2.3 使用規則）: 「本文は最小 18px（OPT-1 強制）」** ⇐ **24px ではなく 18px**
  - L122-124（§2.2 タイプスケール表）: `font.body` = **18px**, `font.body.lg` = 20px, `font.button` 相当 = 18-20px
  - 結果として全モックアップで本文・ボタン 18-20px が実装され、spec.md L13「本文最小 18pt（≒24px）」を**全画面で違反**。これは老眼最適化の根幹（OPT-1）を無効化する重大事象。

#### Major
- **[トークン整合性] components.md と system.md の整合**：
  - components.md §1.3 Button 寸法表で `font.body` (18px) を「md サイズボタン」に割当て。これは system.md §2.2 のタイプスケールには沿うが、spec.md OPT-1 「ボタン文字 18pt（=24px）以上」を満たさない。
  - components.md §1 §4 の ListItem `title` 18pt = 18px 想定。同様。
  - components.md §17 ClockAnswerButtons には「ラベルは数字（『12』『1:30』… と表記）」とあるがフォントサイズ規範未明示。Sprint 3 mockup で 16px 実装される根本原因。**components.md §17 にフォントサイズ最小 18px (or 24px) を明示すべき**。

#### Minor
- **[トークン整合性] system.md §1.4 のコントラスト表における誤計算 4 件**（前述）。表記が AAA だが実測 AA 止まり。表自体を**実測値に書き換え**、システムが「7:1 以上厳守」を実際に守れる palette に再設計するか、要件を「AA = 4.5:1 以上」に緩和してユーザー承認を取る必要。

## 5. 良かった点

- **全 5 mockup × 6 viewport で横スクロール 0 件・コンソールエラー 0 件**：基本的な responsive 実装は堅牢
- **focus-visible 実装が全 mockup で動作**（3px outline + 2px offset、`color.focus.ring` 使用）：NF-9 / system.md §8.1 を満たす
- **ガボール表示領域がダークモードに追従しないこと**を mockup でも実装済み（system.md §1.2 「ガボール背景は OS のダーク化に追従しない」を遵守）
- **設定画面のトグル ON/OFF テキスト併記**（NF-12「色のみで情報を伝えない」）、destructive 行の error 色適用 — 適切
- **screens.md の質**：51 画面の各画面に ASCII 図 + a11y 規範 + 状態遷移 + テスト要件が網羅されている。Planner レビューで「カバレッジ 100%」が出るのも当然のクオリティ
- **components.md 33 コンポーネントの API**：TypeScript 型定義つき、Generator が直接実装に落とせる粒度
- **B-08 全方位改善バッジの進捗ヒント可視化**（Sprint 6 §4）：当初の仕様にない設計上の付加価値、特に観察可能性を高める「3 ゲーム中 N ゲームが先週比で改善中」表記
- **Sprint 5 §3 / §3.1 で「未挑戦」「単体プレイモード」の差分を明確化** — 仕様の曖昧領域を Designer 自ら解消、テスト要件も付与

## 6. 補足: Critical を 3 件と数えた根拠

「Game 3 時計ボタン 16px」は 8 個のボタンで重複検出されるが、「同じ問題群」なので 1 Critical。
「system.md §1.4 のコントラスト誤記」は 4-7 ペアで違反するが、これも「同根の問題」で 1 Critical。
「18pt/18px 混同」は system.md / components.md / 全 mockup に伝播するが、これも 1 Critical。
