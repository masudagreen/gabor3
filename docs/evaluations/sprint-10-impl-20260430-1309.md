## 発見した問題

### [Critical] M-1 修正のリグレッション — `aria-checked` 属性が DOM に一切出力されない

**該当**：Sprint 9 評価レポート (`sprint-9-impl-20260430-1235.md`) で Major 指摘 → Sprint 10 Self-Review §2 で「◯」と申告 → 実機検証で **完全未修正** が判明。

**現象**：
- G-02 プレイ画面で `[role="radio"]` 要素 4 つ（`g02-stimulus-left/right`、`answer-choice-left/right`）を Playwright で取得
- 4 要素すべて、初期状態でも、クリック後でも、再クリック後でも `aria-checked` 属性が **null（属性そのものが DOM に存在しない）**
- 視覚的にはボーダー幅が 0px → 4px、色が黄色（`rgb(255, 197, 61)`）に変化するため、見た目では選択状態がわかる
- だが、SR ユーザーが「左を選んだ」「現在は右が選択中」を認識する手段がない

**根本原因**：
- ImageChoiceCell（`src/components/v11/ImageChoiceCell.tsx`）と AnswerChoiceGroup の ChoiceButton は `accessibilityState={{ checked: isSelected }}` を `Pressable` に渡している
- 一方、`react-native-web@0.19.10` の `createDOMProps`（`node_modules/react-native-web/dist/cjs/modules/createDOMProps/index.js` 行 60〜226）は **`accessibilityChecked`** または **`aria-checked`**（直接指定）を `aria-checked` 属性へ変換するが、**`accessibilityState` オブジェクトはサポートしていない**
- 結果、`accessibilityState.checked` は完全に DOM に伝播しない

**テストが拾えなかった理由**：
- `tests/v11/components/ImageChoiceCell.test.tsx` は `cell.props.accessibilityState` を直接検証する（@testing-library/react-native 経由）
- これは React コンポーネントの props を検証しているだけで、実際に React Native Web が DOM に何を出力するかは検証していない
- そのため、テストは PASS するが実 DOM では aria-checked が欠落する

**再現手順**：
1. `npm run web` でアプリ起動
2. ホーム → 単体プレイ → G-02 → ミニ説明 → はじめる → 距離リマインド 3 秒経過
3. プレイ画面で DevTools の Elements パネルを開く
4. `[data-testid="g02-stimulus-left"]` または `[data-testid="answer-choice-left"]` を選択し属性を確認
5. 期待: `aria-checked="false"`（または `true`）が常に存在
6. 実際: `aria-checked` 属性そのものが DOM に存在しない

**期待される振る舞い**：
- 選択前：全 radio に `aria-checked="false"`
- 「左」をクリック → 左の `aria-checked="true"`、右は `aria-checked="false"`
- 「左」を再クリック → 全部 `aria-checked="false"`
- 「右」をクリック → 右が `aria-checked="true"`、左は `aria-checked="false"`

**修正の方向性**：
- ImageChoiceCell: `accessibilityState={{ checked: isSelected }}` を **削除** し、`Pressable` に `aria-checked={isSelected ? 'true' : 'false'}` を直接渡す（react-native-web 0.19+ では aria-* 属性の直接指定が推奨）
  - もしくは `accessibilityChecked={isSelected}` を併用（deprecated だが現バージョンで動作）
- AnswerChoiceGroup ChoiceButton も同様の修正が必要（こちらは `selected` のみ渡しており checked がそもそもないが、`role="radio"` に対しては `aria-checked` を出力すべき）
- テスト追加：`render` 後に `getByTestId(...).querySelector('[aria-checked="true"]')` などで **実 DOM 属性を検証**するテストを追加（react-native-web 経由のレンダリング結果を見る）。`@testing-library/react-native` ではなく `@testing-library/react`（react-native-web 用）または Playwright での E2E が望ましい
- 影響範囲：G-01 / G-02 双方。G-01 グリッドの 9〜25 個のセル、G-02 の 4 つの選択肢。Sprint 11 以降の G-03 〜 G-13 でも再利用される基盤コンポーネント

**重要度の根拠**：
- a11y 仕様書 NF-10「スクリーンリーダーで主要要素が読み上げられる」を満たさない
- screens.md S10-02 §a11y で「各ボタン `<button role="radio" aria-checked="..">`」と明記されているが満たせていない
- Sprint 9 で Major 指摘 → 修正申告 → 実機未修正 という情報の不整合は、品質保証パイプラインの根本欠陥を示す。テストが「修正済み」を保証しない

### [Major] G-02 ガボールパッチ（ImageChoiceCell）がキーボードで操作できない

**現象**：
- Tab で `[data-testid="g02-stimulus-left"]` にフォーカスは当たり、focus outline 3px は出る
- その状態で Enter または Space を押しても **選択状態が変わらない**（border-width が 0px → 4px にならない）
- 一方、AnswerChoiceGroup の `[data-testid="answer-choice-left"]` は Enter/Space で正しく選択状態が変わる

**再現手順**：
1. G-02 プレイ画面に到達
2. Tab を押して `g02-stimulus-left` にフォーカス（outline 3px が見える）
3. Enter を押下 → 何も起きない
4. Space を押下 → 何も起きない
5. 同じ画面で Tab を 4 回押して `answer-choice-left` にフォーカス
6. Enter を押下 → border 4px に変化（選択される）

**根本原因の推定**：
- ImageChoiceCell の Pressable は子要素として GaborPatch（複雑な div ネスト）を含む
- GaborPatch 内の要素が focusable または key event を intercept している可能性
- AnswerChoiceGroup ChoiceButton は子要素が単純な `Text` のみなので key event が Pressable に届く

**修正の方向性**：
- ImageChoiceCell の Pressable に `onKeyDown={(e) => { if (e.key === 'Enter' || e.key === ' ') { e.preventDefault(); onToggle(); } }}` を web プラットフォームのみで明示追加
- もしくは GaborPatch の root に `pointerEvents: 'none'` または `tabIndex={-1}` を付与し、key event が Pressable まで bubble するように調整

**重要度の根拠**：
- 仕様 §7.2「パッチ自体タップで回答可」は満たしているが、キーボードユーザーは AnswerChoiceGroup ボタン経由でしか回答できない（パッチ直接操作は不可）
- 機能としては代替経路があるため Critical ではないが、a11y 観点で OPT-3「主要操作はキーボードでも到達可能」を部分的に逸脱

### [Major] 1280px / 768px ブレイクポイントでパッチサイズが仕様と乖離

**該当**：screens.md §5「レスポンシブ確認」表

| ブレイクポイント | 仕様 patch | 実装 patch | 仕様 gap | 実装 gap |
|---|---|---|---|---|
| 360px | 100×100 | 100 ✓ | 24 | 72（中央寄せの余白を含む） |
| 375px | 120×120 | 120 ✓ | 32 | 88 |
| 768px | 140×140 | 140 ✓ | 48 | 120 |
| **1280px** | **160×160** | **140** ❌ | 64 | 120 |

**根本原因**：
- `computeG02StimulusLayout()` は `viewportShortSidePx` を引数に取り `shortSide` で判定する
- 1280×800 の場合 `shortSide=800` → 関数の `< 1024` 分岐に入り 140 が返る
- 仕様 §5 のブレイクポイントは「viewport の幅」を意図していると思われるが、実装は「短辺」を使っている
- 768px 横でも shortSide=768（実機テストでは 1024、ただし PC 横の場合）のため、要件と実装で意味が違う

**修正の方向性**：
- screens.md §5 表の意図（幅か短辺か）を明確化
- どちらにしろ 1280×800 では 160×160 が期待されるので、`computeG02StimulusLayout(width, height)` のように両方を考慮するか、`shortSide` ではなく `width` を使う

**重要度の根拠**：
- 視覚体験に大きな差は出ない（パッチが 140 でも 160 でも目視判別は十分可能）
- ただし PC 横画面で「広い画面なのにパッチが小さく感じる」体験劣化があり、screens.md §5 と乖離
