## サマリー

機能の主要動線（ホーム → 単体プレイ → ミニ説明 → リマインド → 60 秒プレイ → 自動採点 → 結果サマリ → 3 ボタンフッター）は G-08 / G-09 とも完全に動作し、回答の選択 / 解除 / 切替・staircase 駆動・Polat 派生 spacing・合成閾値表示「c=0.10 d=3.0λ」も spec §7.8 / §7.9 通りに実装されている。レスポンシブも 360 / 375 / 768 / 1280 で overflow なし。テスト 1356 件 / typecheck 0 errors / build:web 732 kB すべて Generator 自己評価通り。

ただし **GE-08 テストパッチおよび GE-09 中央 target が `ImageChoiceCell disabled` 経由で常時 `opacity: 0.5` で描画される** Critical 1 件を検出。これにより spec §7.9 が要求する「Polat ら 2004/2012 のパラダイム」の核心である target コントラスト値（staircase 0.05〜0.20、初期 0.10）が**実効値で半減**して描画される。staircase が「c=0.10」と記録・表示しても、実際にユーザー網膜に届くコントラストは ~0.05 に減衰しており、spec §7.9 の臨床根拠（GlassesOff 中核訓練、近見視力 +2 行報告）の刺激パラメータ忠実性が破綻している。G-08 でも adapter（不透明度 1.0）と test patch（不透明度 0.5）の間で意図しない明度差が発生し、§7.8 の「上 adapter 高コン 0.6 / 下 test 絶対角度」の関係が崩れる。

修正は ImageChoiceCell 側に `dimOnDisabled` 等の prop を追加するか、G-08/G-09 の stimulus を ImageChoiceCell 経由でラップしないリファクタが必要。同種の問題は G-04〜G-07 では発生していない（SideBySideStimulus は `disabled` を play 中は false として渡すため）ので、Sprint 15 で hard-coded `disabled` にしたのが直接の原因。

## 発見した問題（重要度順）

### [Critical] GE-08 テストパッチおよび GE-09 中央 target が常時 `opacity: 0.5` で描画される（実効コントラスト半減）

**再現手順**
1. `npm run web` で起動、ホーム → 単体プレイ → G-08（または G-09）→ ミニ説明 → 距離リマインド → プレイ画面に到達
2. DevTools Console で次を実行：
   ```js
   const target = document.querySelector('[data-testid="g09-stimulus-target"]'); // G-09 の場合
   // または '[data-testid="g08-stimulus-test"]' （G-08 の場合）
   getComputedStyle(target).opacity
   ```
3. 結果が `"0.5"` となる（プレイ中・回答前後問わず常に 0.5）

**期待される振る舞い**

spec §7.9 通り：
- target コントラストは staircase 値（初期 0.10）そのままが網膜に届く
- 視覚的には flanker（高コン 0.5）より薄いが、それは GaborPatch の正弦波振幅で表現されるべきで、wrapper の opacity で減衰させてはいけない

spec §7.8 通り：
- 下 test patch の振幅は staircase の絶対角度のみで決まる
- adapter（高コン 0.6 / 不透明度 1.0）と test（staircase 角度 / 不透明度 1.0）の不透明度は同一であるべき

**実際の振る舞い**

- G-09: `data-testid="g09-stimulus-target"` の `<div>` に `style="opacity: 0.5"`（getBoundingClientRect で確認）。一方 flanker 左右は opacity 1.0
- G-08: `data-testid="g08-stimulus-test"` の `<div>` に `style="opacity: 0.5"`。一方 adapter は opacity 1.0
- これにより target/test の Gabor パッチ画素が `0.5 × 縞模様 + 0.5 × 親背景(#808080)` で合成され、実効振幅が**半減**
  - 例：staircase contrast 0.10（光度範囲 115〜140 around 128）→ 描画後（121〜134）→ 実効コントラスト ~0.05
  - 表示される「閾値 c=0.10」が実際は c=0.05 相当の刺激での測定結果
- staircase の運動方向は正しく動くが、絶対値が 1 段階分（半分）下に常時シフト

**該当箇所のヒント**

- `src/components/v11/games/VerticalStackStimulus.tsx` L170: `<ImageChoiceCell ... disabled testId="...test">`
- `src/components/v11/games/LateralMaskingStimulus.tsx` L166: `<ImageChoiceCell ... disabled testId="...target">`
- `src/components/v11/ImageChoiceCell.tsx` L125: `opacity: disabled ? 0.5 : pressed ? 0.92 : 1`

**修正の方向性（いずれか）**

A. **ImageChoiceCell にオプション prop 追加（推奨）**
   ```tsx
   // ImageChoiceCell.tsx
   export type ImageChoiceCellProps = {
     ...
     /** disabled 時に opacity 0.5 を適用するか（タップ抑制が目的で視覚減衰が不要なら false） */
     dimOnDisabled?: boolean; // default true
   };
   // L125
   opacity: (disabled && (dimOnDisabled ?? true)) ? 0.5 : pressed ? 0.92 : 1,
   ```
   呼び出し側（VerticalStackStimulus / LateralMaskingStimulus）で `dimOnDisabled={false}` を渡す。テストも更新。

B. **ImageChoiceCell ラップを止める**
   - GE-08 / GE-09 では「タップ可能性」が不要（disabled）なので、ImageChoiceCell の機能（フォーカス枠・拡大ハイライト枠線）を流用する代わりに、`<Animated.View>` + 条件付き枠線 `<View>` で目的を達する。SideBySideStimulus も `disabled={false}` を play 中渡すため、こちらは無関係。

C. **ImageChoiceCell の `disabled` 自体を見直し**
   - "disabled" の本来意味（タップ抑制）と "視覚的に薄く" は分離すべき。Sprint 16 以降の GE-11 Vernier や GE-12 クラウディングでも同じ罠に嵌まる懸念があるので、A 案で根本修正が望ましい。

いずれの修正後も `tests/v11/components/games/{Vertical,Lateral}*.test.tsx` の「disabled（タップ不可）」アサーションは維持しつつ、**「opacity が 1 である」アサーションを追加**することを推奨（DOM レンダリング後の computed style で確認）。

### [Minor] プレイ中ユーザーが回答選択した時点で test/target に黄ハイライト枠（4px）が描画される

**再現手順**
1. G-09 のプレイ中、「縦寄り」または「横寄り」を選択
2. 中央 target に黄色（#FFC53D）の 4px 枠が表示される

**期待される振る舞い**

screens.md S15-05 / S15-06 の記述では、黄拡大ハイライトは**採点後の正解開示時**のみ。

**実際の振る舞い**

`isTestSelected = selectedDirection !== null || highlightDirection != null;` のため、ユーザー選択中も `isSelected=true` となり、ImageChoiceCell の `borderWidth: isSelected ? 4 : 0, borderColor: isSelected ? colors.highlightCorrect`（黄）で枠が描画される。

**該当箇所のヒント**

- `src/components/v11/games/VerticalStackStimulus.tsx` L118: `const isTestSelected = selectedDirection !== null || highlightDirection != null;`
- `src/components/v11/games/LateralMaskingStimulus.tsx` L114: 同等

**修正の方向性**

`isSelected` を `highlightDirection != null` のみで判定（ユーザー選択中は枠なし）。回答選択のフィードバックは下の AnswerChoiceGroup ボタン側の selected 状態で十分にユーザーに伝わる。spec §7.8 / §7.9 の本来意図（"画像切替なし、フェードなし、点滅なし"）にも、刺激領域のラジカル変化を入れない方が忠実。

ただし、これは UX 改善観点の Minor。Critical の修正の "ついで" に直すと工数効率良し。

### [Minor] G-08 / G-09 とも `accessibilityElementsHidden` が RN-Web で `aria-hidden` に変換されない（プラットフォーム制約）

**再現手順**
1. Playwright の accessibility snapshot で adapter / flanker を確認
2. 「上の adapter（読み上げ対象外）」「左の flanker（読み上げ対象外）」が generic ノードとして読み上げ可能状態にある

**実際の振る舞い**

`accessibilityElementsHidden` + `importantForAccessibility="no-hide-descendants"` は React Native でしか効かず、RN-Web では DOM 上の `aria-hidden` に翻訳されない（Sprint 9〜14 全てで同様の状況。プラットフォーム既知制約）。

ariaLabel に「読み上げ対象外」と明示してあるため、SR ユーザーがそれを聞いて文脈把握はできる。Critical/Major には該当しない。

**修正の方向性（任意）**

将来的に `Platform.OS === 'web'` で `aria-hidden="true"` を直接付与するヘルパーを ImageChoiceCell / accessibilityProps に追加。Sprint 15 単独の問題ではないので**本スプリント修正対象外**でよい（プロジェクト横断課題）。

## 良かった点（次回も維持してほしい）

1. **Polat パラダイムの派生実装が秀逸**：`derivePolatSpacingFromContrast` で contrast → spacing の 3 サンプリング点（0.05/1.5λ、0.10/3λ、0.20/5λ）を線形補間する純関数化、staircase の単一 contrast 軸更新で flanker 距離が連動する設計は spec §7.9 の意図を素直に実装。
2. **合成閾値表記「c=0.10 d=3.0λ」**：MetricCard の主要値に 2 行表記で表示する `formatG09CombinedThresholdLabel` の発想が良い。screens.md S15-06 と一致。
3. **`computeG09SpacingPx` の viewport クランプ**：staircase 進行で flanker 距離が広がっても画面幅に応じて自動クランプ。360 / 768 でレイアウト破綻なしを実機確認済み。
4. **テスト 210 件追加**：lib / components / screens 各層に網羅的に追加（受け入れ +30 を 7 倍）。Polat 派生関数も単独テスト。
5. **3 ボタンフッター（同じゲームをもう一度 / ゲーム一覧へ戻る / ホームへ）**：Sprint 9〜14 と同一構造で UX 一貫性。
6. **未回答 → 不正解 + staircase 易方向クランプ**：G-08 / G-09 とも実機で確認（staircase G-08: 5 → 6、G-09: 0.10 → 0.11、spec §7.8 / §7.9 通り）。
7. **アクセス可能性**：Tab フォーカス → 3px 青 outline、Space で aria-checked 動的トグル、再タップで解除を実機で検証。

## 不合格判定の根拠

Critical 1 件のみだが、これは spec §7.9 の冒頭で強調される「Polat ら 2004/2012 の Lateral Masking パラダイム」の臨床的根拠（GlassesOff 中核訓練、近見視力 +2 行）を支える刺激パラメータの**忠実性**そのものが破綻するため、本スプリントの最重要受け入れ基準（「Polat ら 2004/2012 の Lateral Masking パラダイムが機能する」）が**機能はしているが正しい刺激値で機能していない**。spec の根拠を支えるためには修正必須。
