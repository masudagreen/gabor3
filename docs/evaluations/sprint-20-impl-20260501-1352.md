## 発見した問題（重要度順）

### [Critical] ResultOverlay が MarkBadge を刺激パッチに重畳していない（targetId 生文字列の露出）

**再現手順**：
1. http://localhost:8083/ → 単体プレイ → G-04（コントラスト弁別） → 「はじめる」 → 任意の選択（例：「左が濃い」）
2. 60 秒経過まで待つ（もしくはどの単数選択ゲームでも同じ）
3. 結果オーバーレイが表示される

**期待される振る舞い**（spec-v11.md F-10 v1.1.1 / screens.md §2.5 / §4.1 / §4.2）：
- 正解選択肢（例：右パッチ）の**中央**に緑 ◯ アイコンが重畳される
- ユーザー誤選択（例：左パッチ）の**中央**に赤 ✕ アイコンが重畳される
- ◯/✕ サイズはセルの 35〜40%（`markBadgeSizeForCell` ヘルパに準拠）
- ガボール本体の縞模様を完全には覆い隠さない（不透明度 82% の背景）
- ユーザーが操作した対象と結果が**空間的に対応**することがこの UX の核心（screens.md §2.5）

**実際の振る舞い**：
- 刺激領域（左右の `extra-stimulus` ガボール）の**下方**に、別レイヤーとして以下のテキストリストがレンダリングされている：
  ```
  ◯  g04-right
  ✕  g04-left
  ```
- ガボール本体の上には何も重畳されていない
- ユーザーには `g04-right` / `g04-left` という意味不明な内部 ID 文字列が露出する（screens.md にはこのような表示要件は皆無）
- 単体プレイでは特に「次へ」ボタン直下にこのリストが残るため、ユーザー体験の核心 UX（「ここを選んだ → ここが正解」の空間的対応）が**完全に欠落**

**該当箇所のヒント（ファイル / セレクタ）**：
- `src/components/v11/ResultOverlay.tsx` 行 268-309
  - 行 268-272 のコメント自身が「**まずはインラインで全 marks を一覧表示する**」「Web 実機では絶対配置版が別途絡む形を想定するが、本実装ではセル直近の data-target-id 検索 + position:absolute 配置を『ResultOverlayPortal』（後述任意機能）として親が担う設計とする」と明示しており、**重畳機能を意図的に未実装で残してある**
  - 行 286-309 の `marksContainer` ブロックで `<View styles.markRow>` 内に `<MarkBadge>` + `<Text>{m.targetId}</Text>` を縦に列挙する形になっている
- `ResultOverlayPortal` という名前で grep しても、コメント以外の実装は **src/ ツリー内に存在しない**
- DOM testID `result-overlay-marks` の中身がそのままユーザー可視のリストとして残っている

**修正の方向性**：
1. `ResultOverlay` に Web 専用の重畳ロジックを追加：
   - `useEffect` で `marks` 受領後、`document.querySelector('[data-target-id="..."]').getBoundingClientRect()` を 13 ゲーム分マウント時のみ計算
   - 各 MarkBadge を `View` でラップし、`Platform.OS === 'web'` 時に `position: absolute` + `top` / `left` で刺激領域内に絶対配置
   - 刺激領域全体に対する相対座標が必要なので、`ResultOverlay` 自体が ScrollView 内ではなく、`extraStimulus` を相対座標の親として持つ構造にリファクタが必要
2. または「ResultOverlayPortal」設計を実装：刺激側の各セル `<View>`（例：`SideBySideStimulus` の `ImageChoiceCell` ラップ）が `result.phase` 中に MarkBadge を内側に absolute 重畳できるよう、`overlayMark?: ResultMark` props を全 stimulus 系コンポーネントに追加し、刺激側がレンダリング責務を担う形に変更
3. `marksContainer` から targetId 文字列の `<Text>` 表示は **完全に撤去**（ユーザーには見せない）。テスト用には `data-target-id` 属性付き不可視 `<View>` でも置き換え可能
4. 13 ゲーム分の `extraStimulus` レイアウトと整合性を取る必要あり：
   - G-01 / G-07：4×4 / 4×4 グリッドの各セル中央
   - G-02 / G-04 / G-05 / G-06 / G-09 / G-11：左右 2 セル中央（または answer-choice ボタン中央）
   - G-03：時計 8 ボタン中央
   - G-08：下左右 2 テストパッチ中央
   - G-10：4 象限ボタン中央
   - G-12：horizontal-4 アイコン中央
   - G-13：keypad-10 ボタン中央
5. 修正後、新規 e2e（Playwright）テストで「`getBoundingClientRect` 上、MarkBadge の中心が刺激セル中心の ±20px 以内」を検証する Playwright assertion を追加することを強く推奨（同種のリグレッションを未然防止）

> **テスト 2249 件全 PASS であっても、この Critical を検出できなかった理由**：既存テストは `getByTestId('result-overlay-mark-badge-${targetId}')` の DOM 存在のみ検証しており、**実 DOM 上の絶対座標による「刺激セル中心への重畳」を一切検証していない**。Self-Review §3.7 の「単体プレイ動線：60 秒経過時に同じ Screen 内に `result-overlay-action-bar` が表示される」も同様で、**結果開示の「位置」までは見ていない**。

### [Minor] G-02 / G-08 の guidance に `role="status"` / `aria-live="polite"` が無い
screens.md §3.6（G-02）／§5.6（G-08）では「guidance：`role="status"`, `aria-live="polite"`（出題方向が変わるたびに読み上げ）」と規定されている。実装の guidance テキストは `<div>`（24px）で出力されており、`role` / `aria-live` 属性が無い。

ただし、設問の出題方向（cw / ccw）は試行ごとに rng で決まるが、画面遷移後はその試行内では一定であり、画面表示中の動的更新は発生しないため、SR が読み上げを逃すケースは限定的。screens.md 規範を満たすには属性追加が望ましいが、ユーザー体験への実害は限定的なため Minor。

**修正方向性**：`G02SideBySideTiltScreen.tsx` / `G08TiltAftereffectScreen.tsx` の guidance `<View>` に `accessibilityRole="status"` + `accessibilityLiveRegion="polite"` を付与。

### [Minor] G-04 のパッチが `role="radio"` 化されているが、horizontal-2 ボタンと併用で重複選択肢が出現
G-04 のプレイ画面で、左右ガボールパッチが `role="radio"` + 動的 `aria-label="左の縞模様（タップで「左が濃い」と回答）"` で**直接タップ可能**になっている。しかし spec §7.2 G-02 / §7.8 G-08 のみが Sprint 20-C で「直接選択化」が要求対象であり、G-04（コントラスト弁別）は要求対象外。

これは **`SideBySideStimulus` の `stimulusInteractive=true` を 13 ゲーム共通で有効化した波及効果** と推察される（Self-Review §1.4「G-02 / G-08 のように『刺激領域が選択肢を兼ねる』ゲームでは true にしないと SR / 自動テストが radio に到達できない」とあり、stimulusInteractive を共通有効にした可能性）。

ユーザー体験への害は限定的（むしろ操作性向上）だが、SR には「左の縞模様」「左が濃い」の 2 通りで同じ選択肢が読み上げられる重複が生じる可能性がある。`stimulusInteractive` は G-02 / G-08 のみで true、他は false にするのが規範通り。

**修正方向性**：`AppRouterV11` または `G04ContrastDiscrimScreen.tsx` で `SideBySideStimulus` 呼び出し時に `stimulusInteractive={false}`（既定）を明示。G-02 / G-08 のみ true。

### [Minor] React Native Web の `props.pointerEvents is deprecated` 警告
ブラウザコンソールに 1 件の警告：
```
[WARNING] props.pointerEvents is deprecated. Use style.pointerEvents
```

これは ResultOverlay または既存 stimulus の `pointerEvents="none"` props 指定に由来する RN Web の deprecation 警告で、過去スプリントから継続している既知の現象。動作には影響しないが、Sprint 21 以降で `pointerEvents` を `style={{ pointerEvents: 'none' }}` に書き換えることが望ましい。
