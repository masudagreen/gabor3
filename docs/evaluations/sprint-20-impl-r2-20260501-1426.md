# 評価レポート: Sprint 20 実装評価ラウンド 2（v1.1.1 結果開示と選択 UX 改善）
日時: 2026-05-01 14:26
判定: ❌ **不合格（Critical 1 / Major 0 / Minor 1）**

## サマリー

ラウンド 1 Critical（テキストリストとして targetId を生表示する旧 marksContainer）と Minor 1（G-02 / G-08 guidance の role=status / aria-live）は**真正修正**された。**G-02 / G-08** では MarkBadge が刺激パッチ中央へ完全に絶対重畳され、targetId 文字列の露出も無く、F-10 v1.1.1 の核心 UX が機能している（実機 1280 / 360 両方で中心座標差 0px、サイズ 30〜37%、components.md §24 規範通り）。

しかし、**重畳描画は 13 ゲーム中 G-02 / G-08 の 2 ゲームでしか機能していない**。他の 11 ゲーム（G-01 / G-03 / G-04 / G-05 / G-06 / G-07 / G-09 / G-10 / G-11 / G-12 / G-13）では結果オーバーレイの extraStimulus 内にも document 全体にも `data-target-id="g04-left"` 等に対応する要素が存在せず、ResultOverlay の重畳探索がすべて失敗 → 全 MarkBadge が fallback（off-screen, opacity=0）に配置される。実機画面に ◯/✕ アイコンが表示されない（G-04 / G-01 結果スクショで確認）。

これは ResultOverlay 自身の重畳ロジック実装は正しい（G-02 / G-08 で完璧に動作）一方で、**他 11 ゲームの screen 側で `SideBySideStimulus` / `ImageChoiceCell` / `AnswerChoiceGroup` 等に渡す `dataTargetId` / `leftDataTargetId` / `rightDataTargetId` 等の prop が漏れている**ため、`resultMarks.ts` が生成する `g04-left` / `g01-r0c0` / `g13-key-3` 等の targetId と DOM 上の属性が一致せず重畳できない構造的不備。

旧 Critical（targetId 文字列の生露出）と比べると一段進歩したが、F-10「ガボール／選択肢の中央に ◯/✕ アイコンを重畳。自分が選んだ場所と結果が空間的に対応する UX」という核心要件は依然として 11/13 のゲームで満たされていない。

> **テスト 2258 件全 PASS であっても、この Critical を検出できなかった理由（再掲）**：既存テストは `getByTestId('result-overlay-mark-badge-${targetId}')` の DOM 存在のみを検証しており、**fallback 領域に置かれた MarkBadge も「DOM 上に存在する」と判定されて通過してしまう**。Web 実機での `getBoundingClientRect` を使った位置検証が必要。

- G-01：結果画面の刺激プレビュー領域そのものが空、MarkBadge 5 個全て fallback（off-screen, opacity=0）

**該当箇所のヒント**：
- `src/components/v11/games/SideBySideStimulus.tsx` L99-205：`leftDataTargetId` / `rightDataTargetId` props のデフォルトが `g02-left` / `g02-right`。**G-04 / G-05 / G-06 / G-09 / G-11 が SideBySideStimulus を再利用するときにこの props を渡していないため、すべてのゲームで data-target-id が `g02-*` 固定になる**
- `src/screens/v11/games/G04ContrastDiscrimScreen.tsx` L327-343：`SideBySideStimulus` 呼び出しで `leftDataTargetId="g04-left"` / `rightDataTargetId="g04-right"` を**渡していない**

- `src/screens/v11/games/G13EmbeddedDigitScreen.tsx`：`AnswerChoiceGroup` の各 keypad ボタンに `g13-key-${digit}` を渡す必要
- `src/components/v11/ResultOverlay.tsx` L262：`document.querySelector('[data-target-id="..."]')` がドキュメント全体を検索するように既にフォールバック実装あり。だが targetId そのものが DOM に出ていないため救済されない
- `src/lib/v11/resultMarks.ts` L37 / L56 / L74 / L94 / L114 / L151 / L185 / L204 / L223 / L242：marks の targetId はゲームごとに正しく一意命名されている。逆に screen 側がこの命名を尊重して DOM に伝搬する必要がある

**修正の方向性**：

各 PlayScreen / 結果プレビュー側で、resultMarks.ts の出力 targetId と一致する `data-target-id` 属性を付与する。

| ゲーム | 修正対象 | 渡すべき targetId |
|---|---|---|
| G-01 | `ImageChoiceCell` の各セル（プレイ画面 + 結果プレビュー両方） | `g01-r${row}c${col}` |
| G-03 | 時計 8 ボタン | `g03-clock-1` 〜 `g03-clock-8` |
| G-04 | `SideBySideStimulus` の left / right props | `g04-left` / `g04-right` |
| G-05 | `SideBySideStimulus` の left / right props | `g05-left` / `g05-right` |
| G-06 | `SideBySideStimulus` の left / right props | `g06-left` / `g06-right` |
| G-07 | グリッド各セル | `g07-${id}` |
| G-09 | `SideBySideStimulus` の left / right props（または刺激パッチ） | `g09-${orientation}` |
| G-10 | 4 象限ボタン | `g10-${quadrant}` |
| G-11 | `SideBySideStimulus` の left / right props | `g11-left` / `g11-right` |

加えて：
- **結果オーバーレイの extraStimulus が「ゲームごとの刺激プレビュー」を描画する責務**を負う。G-01 では 4×4 グリッド復元（または刺激領域の縮小コピー）を `data-target-id` 付きで extraStimulus に出力する必要あり。現状 G-01 の結果画面で extraStimulus が空のため、たとえ targetId が一致しても重畳先が無い。
- ResultOverlay は document 全体に fallback querySelector するため、プレイ画面の DOM が結果オーバーレイ表示中も維持される構造（同じ Screen の子として ResultOverlay が描かれる）であれば、プレイ画面の data-target-id を借用できる。仕様書 F-10 の「同じ Screen 内 result phase」と整合。

リグレッション再発防止のため、Playwright e2e テスト or `getBoundingClientRect` を使う JSDOM テストで「全 13 ゲームの結果オーバーレイで MarkBadge の中心が target セル中心の ±20px 以内」を検証する assertion を追加することを強く推奨。

### [Minor] React Native Web の `props.pointerEvents is deprecated` 警告がコンソールに 9 件残存

**再現手順**：任意のページで F12 → Console
**期待**：警告 0 件（Generator が 9 ファイル / 10 箇所を style 形式に書き換え済みと報告）

**修正の方向性**：`React.useEffect(() => { console.warn = original }, [])` のラッパで stack trace 取得 → 発生源特定 → 該当箇所の View 呼び出しを修正。Sprint 21 以降で対応可
