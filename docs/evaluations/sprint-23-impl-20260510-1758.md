## 良かった点

- **Native プレイヤー集約設計**: tick3/tick2/tick1 を同じ `tick.mp3` キーに集約し、createAudioPlayer 呼び出しを 7 → 5 回に削減。GC 負荷も最小化。プレイヤー再利用 (`seekTo(0) → play()`) で連続再生しても新規インスタンス生成しないのは expo-audio 公式推奨パターンに沿う
- **iOS / Android で `setAudioModeAsync` の引数を出し分け**: rapidreading2 で報告された Android Enum 解決問題を回避し、iOS だけに完全モード、Android は `shouldPlayInBackground: false` のみ。実運用知見の継承が見事
- **NF-33 サイレントモード尊重を `playsInSilentMode: false` で OS 任せ**: アプリ側でフラグ管理する設計を避け、iOS の Mute Switch を OS 層で尊重。ハプティクスは別経路で継続発火。仕様準拠と実装シンプルさを両立
- **静的 require の switch dispatch**: Metro の静的解析制約を理解し、5 ケース手書きで対応。これが将来「音源を増やす際の追加点が明示的」というドキュメント効果も持つ
- **テストの厳密性**: `jest.mock('expo-audio')` で `__mockPlayers` / `__mockSources` を捕捉する設計が後続改修時の retrocompatibility 検証にも流用可能
- **Web bundle 不変 (796 kB)**: Native 専用 require が Web bundle に一切漏れないことを実機 grep + ネットワーク 2 経路で確認済み

## 申し送り

Native 実機 (iOS / Android Expo Go) での以下の項目はユーザー手動 QA に委ねる：

1. **音再生の体感**: correct/wrong/tick/end/badge.mp3 が実際に鳴り、音量バランスが妥当か
2. **NF-33 サイレントモード**: iOS 物理デバイスでサイレントスイッチ ON にした際、音が鳴らずハプティクスのみ継続することを目視確認
3. **NF-31 レイテンシ (100ms)**: 初回 1 発目の遅延 (lazy 生成のため理論的には起こり得る)。問題があれば PostSession で prime する追加実装を検討
4. **連続再生安定性**: tick.mp3 が残り 3-2-1 秒で 3 連発正しく鳴るか

これらは Generator も Evaluator も実機を持たないため自動化不可、ユーザー実機確認後に
v1.2 リリース前最終 GO/NO-GO を判断する想定。
